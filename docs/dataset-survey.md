# 真实数据集调研与首批接入方案

## 1. 目标与约束

当前阶段我们不是追求大规模 SOTA，而是验证以下工程闭环是否稳定：

1. 真实视频 `->` 统一抽帧格式；
2. 条件 `c` `->` 兼容现有 8 维离散条件接口；
3. 能在真实数据上跑通 `recon + dyn + cond` 的短训；
4. 产出可复现的第一批评估结果。

因此首批数据更看重：

1. 公开可获取；
2. 体量适中，下载和预处理成本低；
3. 有类别信息，可映射为当前条件接口；
4. 不需要复杂授权流程。

## 2. 候选数据集对比（用于 MVP）

### UCF101

优点：

1. 经典公开视频动作数据，社区认知高；
2. 类别标签清晰，便于做条件桥接；
3. 有多个公开镜像和子集，适合先跑小批次。

风险：

1. 完整版体量较大；
2. 官方原始分发格式和下载链路对自动化不够友好。

### HMDB51

优点：

1. 同为经典动作数据；
2. 语义类别明确。

风险：

1. 体量与整理成本仍不小；
2. 对我们当前 MVP 的“快速闭环”不如小型子集友好。

### KTH Actions

优点：

1. 数据相对规整；
2. 规模较小于大型现代数据集。

风险：

1. 场景多样性偏弱；
2. 对“真实复杂分布”覆盖有限。

### Something-Something（V2）

优点：

1. 时序与条件变化信息丰富。

风险：

1. 体量与数据管理复杂度较高；
2. 不适合当前快速 MVP 迭代节奏。

## 3. 首批接入选择

首批采用：`sayakpaul/ucf101-subset`（Hugging Face 上公开子集，约 171MB）。

原因：

1. 规模足够小，适合快速拉通；
2. 来源公开，可复现下载；
3. 仍保留真实视频内容与类别标签；
4. 便于后续无缝切换到更大规模 UCF101/HMDB51。

## 4. 条件桥接策略（MVP 工程版）

现有训练代码要求每个样本提供 8 维离散条件：

`shape, color, dir_x, dir_y, size, speed, motion, background`

真实动作数据不天然给这 8 维标签，因此首批接入采用“稳定哈希映射”作为桥接：

1. 以类别名 `label` 为输入；
2. 对每个条件键做独立哈希取模；
3. 得到满足各自基数约束的离散值；
4. 保证同一类别跨运行映射一致（固定 seed 与哈希规则）。

这不是最终语义标注，只是用于验证训练链路、损失行为与工程可行性。

## 5. 下一步升级方向

1. 从“哈希桥接”升级到“可解释条件提取”（视觉属性、文本标签、伪标注器）；
2. 引入更强的时间采样策略（随机片段、多视图）；
3. 分阶段迁移到更大真实数据集；
4. 增加真实数据上的定量指标与可视化分析模板。

## 6. 实验记录位置

后续测试结果不再继续堆积在本调研文档中，而统一记录到：

1. `runs/TEST_INDEX.md`：测试总清单；
2. `runs/<run_name>/log.md`：对应实验的独立日志。

这样可以把“数据调研”和“实验结果”拆开，避免后续文档混杂。

## 7. 当前已知限制

1. 条件来自类别哈希桥接，不是语义可解释标注；
2. 真实数据上的条件监督仍偏弱；
3. 当前结果首先服务于链路验证，而不是最终性能评估。
