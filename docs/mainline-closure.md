# VH-01 主线闭环说明

## 1. 文档目的

本文用于把当前阶段的理论对象、工程对象、实验验证与阶段性结论统一串成一条闭环主线。

它不是完整理论文档的替代品，也不是实验日志的替代品，而是回答一个更直接的问题：

> 到当前为止，VH-01 的主线到底在工程上实现到了哪一步？

## 2. 理论对象到工程对象的映射

### 2.1 潜轨迹与内部结构

理论上，我们讨论的是潜轨迹 `Z_{1:T}`、残差张量 `R_{i,j,l}` 与动态一致性结构 `E_dyn`。

在当前工程中，对应关系是：

- `Z_{1:T}` 对应编码器得到的潜序列
- `R_{i,j,l}` 对应多起点、多跨度 rollout 与真实潜轨迹/观测之间的残差
- `E_dyn` 对应训练时的动态一致性与预测误差项

因此，当前底座已经具备了“在潜轨迹层面讨论内部动态结构”的最小实现。

### 2.2 条件更新与测度代理

理论上，更准确的链条应当写成：

$$
\mu_0 \xrightarrow{\mathfrak T_c} \mu_c \xrightarrow{\mathrm{Read}} \Gamma
$$

其中：

- `\mu_0` 是训练阶段隐含赋予全体可能轨迹的先验测度；
- `\mathfrak T_c` 是条件 `c` 诱导的测度更新或尖锐化算子；
- `\mu_c` 是更新后的测度；
- `Read` 才是后续的 `top-1`、`top-k`、高支持区域读取。

当前工程里，还没有显式实现这条链上的连续对象。现有代码更准确地说，只实现了若干离散代理：

- [mvp.py](/home/void0312/AIGC/VH-01/src/vh_mvp/models/mvp.py#L227) 的 `condition_candidate_logits` 给出的是有限条件目录上的离散打分，可视为“更新后质量分布”的一个离散代理切片；
- [mvp.py](/home/void0312/AIGC/VH-01/src/vh_mvp/models/mvp.py#L217) 的 `condition_alignment_energy` 给出的是 rollout 族相对于查询条件的对齐能量，可视为“条件更新作用”的规划侧代理；
- [support.py](/home/void0312/AIGC/VH-01/src/vh_mvp/support.py#L19) 到 [support.py](/home/void0312/AIGC/VH-01/src/vh_mvp/support.py#L145) 的 `CandidatePosterior`、`CandidateSet` 与 `query_responsive_selection` 属于读取层，它们处理的是离散代理上的排序、截断和选择，而不是理论本体。

因此，当前更准确的映射不是“理论里的候选集已经被完整实现”，而是：

1. 潜动态结构已实现；
2. 条件更新的离散代理已实现；
3. 更新后分布的读取规则已实现；
4. 但 `\mu_0`、`\mathfrak T_c`、`\mu_c` 仍未被显式对象化。

## 3. 当前读取规则（工程近似）

由于当前代码里并没有一个统一、显式、可直接调用的 `\mu_c`，所以现阶段必须额外写一个读取规则，把两个离散代理拼接成可执行决策。

当前采用的工程近似不是：

- 直接把 `support_top1` 当作最终输出；
- 也不是把联合打分误当作理论中的更新算子；

而是定义统一读取集合：

$$
\mathcal T_{\mathrm{exec}}^{\alpha,\beta}(x,c)=T_{obs}^{\alpha}(x)\cap C_{plan}^{\beta}(c)
$$

其中：

- `T_obs^\alpha(x)` 是观测后验候选集
- `C_plan^\beta(c)` 是查询条件诱导的规划核心

若交集为空，则回退到 `C_plan^\beta(c)`。

最终选择规则为：

$$
\hat i(x,c)=\arg\max_{i\in \mathcal T_{\mathrm{exec}}^{\alpha,\beta}(x,c)} p_{obs}(i\mid x)
$$

这条规则实现于 [support.py](/home/void0312/AIGC/VH-01/src/vh_mvp/support.py#L123) 的 `query_responsive_selection`，但它的准确定位应当是：

> 当前离散代理缺乏统一 `\mu_c` 时的读取层补丁。

它不是理论中的核心算子，只是理论对象尚未显式化之前的工程近似。

## 4. 为什么这条规则只能算当前工程补丁

这条规则之所以仍然值得保留，是因为它在当前实现条件下有明确用途：

1. 查询条件先决定“允许在哪个局部核心中选”
2. 观测后验只负责在该局部核心内部做可行性排序

它避免了两种已经被实验排除的极端：

- 只按查询支持最高项直接选择
- 在全局上把观测后验与查询后验简单相乘

前者缺乏观测可行性，后者过于容易退化成观测端主导。

但它的边界也必须说清楚：

- 它不是 `\mathfrak T_c`；
- 它不是 `\mu_c`；
- 它甚至不是理论核心，只是 `Read[\mu_c]` 在当前离散代理条件下的一种替代实现。

## 5. 实验证据

### 5.1 读取层闭环已成立

当前已经完成：

- 离散后验与读取对象正式化
- 最小读取约束 rollout
- 条件扰动实验
- 读取约束生成对照
- 第一轮读取规则研究
- 查询响应读取规则接入执行链
- 查询响应读取规则接回训练主线

### 5.2 `support_refined` 是当前最适合作为执行底座的版本

支持评测上，`support_refined` 相比 `aligned`：

- 真实条件概率更高
- 支持宽度更小
- 候选集更紧

虽然覆盖略有回落，但它更适合作为真正可操作的离散读取底座。

### 5.3 查询响应读取规则已经具有明确作用

在 `support_refined`、`alpha=0.90`、`obs_alpha=0.90`、`plan_core_alpha=0.50` 下：

- `true_direct = 0.046843`
- `true_top1 = 0.048260`
- `true_query_responsive = 0.047052`
- `true_set_best = 0.044006`

这说明：

- 查询响应读取规则显著优于 `support_top1`
- 它还没有达到 oracle 水平
- 但它已经是一个正式、可执行、结构上更合理的读取补丁

从响应性看，在 `support_refined` 上：

- `far query` 下，`query_responsive` 对查询条件的匹配率为 `20.0%`
- 同时它与 `obs_top1` 相同的比例降到 `32.5%`

这比简单联合规则明显更符合“查询响应”的目标。

### 5.4 查询响应读取规则已经进入训练主线

在最新的 `query_mainline` 运行中，query-aware 指标已经被正式写回训练主线，而不再只是分析脚本里的事后统计。

这一轮并没有改变梯度本身，但它把以下对象第一次统一放到了训练轨迹里：

1. 支持收缩；
2. 查询响应执行质量；
3. 回退率；
4. checkpoint 选择。

结果很关键：

- `best.pt = epoch=7`，对应“宽集合、零回退”的保守点；
- `best_recon.pt = epoch=8`，对应执行质量最好的折中点；
- `best_support.pt = epoch=9`，对应更尖、更查询驱动、但回退更高的点。

这说明：

1. query-aware 规则已经不只是分析工具，而是正式训练观测量；
2. 但当前的 query-aware checkpoint 准则仍然偏向“零回退宽集合”；
3. 下一阶段需要先解决的是目标平衡问题，而不是重新证明闭环是否成立。

这一问题随后已经在 `query_balanced_mainline` 中被正式修正：

- 新的 `best.pt` 已稳定落在 `epoch=8`
- 它在 `true query` 下达到 `query_responsive_mse = 0.046689`
- 同时保持 `true fallback_rate = 0.0`

因此，当前默认正式执行点已经不再是上一轮那个过宽的保守点，而是“执行优先、预算内低回退”的平衡点。

## 6. 当前可以成立的阶段性结论

当前我们可以严格地说：

1. VH-01 已经不再只是一个表征学习 MVP。
2. 它已经具备了“离散条件打分代理 -> 读取规则 -> 统一执行集合 -> 最终选择”的最小闭环。
3. 查询响应读取规则已经不是临时分析产物，而是正式工程对象。
4. 查询响应读取规则现在也已经进入训练主线，而且默认 checkpoint 的错位问题已经被修正。

但我们也必须保持清楚：

1. 这还不是最终强生成系统。
2. 这还不意味着生成质量已经最优。
3. 这也还不意味着理论已经被完全实现。
4. `\mu_0`、`\mathfrak T_c`、`\mu_c` 在工程里仍未被显式对象化。
5. 当前完成的主要是读取层闭环，而不是理论本体的完全闭环。

## 7. 阶段切换判断

基于以上结果，可以认为：

> VH-01 当前阶段的读取层补全任务已经基本完成，但理论核心映射仍需继续收紧与显式化。

这里的“下一阶段”不是回去继续无目的调参，而是：

1. 重新收束“先验测度代理 / 条件更新代理 / 更新后测度代理 / 读取规则”之间的对象映射
2. 检查现有训练目标里哪些部分真正对应 `\mu_0` 与 `\mathfrak T_c`，哪些只是读取层补丁
3. 在此基础上，再决定是否扩展 query-aware 观测、扩展数据范围或把相关目标写回训练损失

因此，下一阶段已经不是“先把底座补出来”，而是：

> 在读取层已经闭环且默认 checkpoint 已修正的底座之上，回到理论核心映射本身，重新检查哪些对象真正已被实现、哪些还只是离散代理。
